# server/ — the content platform root

This folder is what a community server will eventually own and distribute:
its config plus every piece of game content it serves. The client reads the
same layout locally today (singleplayer is "your own local server" in
spirit), which is why the folder ships in the repo.

```
server/
├── server.cfg           # dedicated server config: name, max players, tick rate
└── content/
    ├── weapons/         # WeaponDef files: id, kind, damage, range, cooldown
    ├── entities/        # EntityDef files: pickups, bots, props (size/solid/
    │                    #   carryable/max_health/respawn_seconds/visual parts)
    ├── items/           # reserved: non-weapon items (nothing here yet)
    ├── maps/            # reserved: world-type geometry is code-defined today
    │                    #   (src/shared/world_template.cpp) - see World types
    └── sounds/          # original synthesized WAVs (mono 16-bit) + CREDITS.md
```

Rules of the format:

- One definition per file, flat `key = value` lines, `#` comments — the same
  parser as every other TacMove config (`src/shared/content_loader.cpp`).
- Definitions are looked up by their stable string `id`. World saves store
  ids only, so editing a definition retunes every world that uses it.
- The client loads this folder at startup; a definition with the same id as
  a C++ builtin replaces it. If the folder is missing the builtins keep the
  game runnable (see `ContentRegistry::registerBuiltins`).
- `respawn_seconds` is the respawn rule: 0 = never comes back, and 0 is the
  default for EVERYTHING (pickups, bots, props alike - taken or destroyed
  means gone). Set it >0 in a def file, or per placed entity in a world
  save, to opt into respawning.
- `carryable = 1` marks light props the player can E-carry; heavy or static
  objects leave it 0. This is deliberately a boolean, not a weight system.
- `gear = 1` marks equipment pickups (the shield): E equips them instead of
  granting a weapon or starting a carry.
- Weapon defs can set `weight` (fists 0.6 … sword 1.0): it scales melee
  windup/recovery times and stamina costs in `src/shared/melee.cpp`.

World types (the starting layout a new world is created from) are defined in
`src/shared/world_template.{h,cpp}` as pure data: a `WorldTemplate` holds an
id, display name, description, static geometry, spawn point, and starting
entity placements. `world.cfg` stores the chosen `world_type` id; old saves
without one fall back to `flat_sandbox`. The five built-in types are
`flat_sandbox`, `duel_yard`, `aim_range`, `movement_course`, and
`social_hub`. The struct is shaped so a future loader can fill it from
`maps/*.cfg` without touching callers.

Still hard-coded in C++ (documented seams, not accidents):

- Map geometry lives in `world_template.cpp` (the `maps/` data-file loader
  is the next step; the template struct is already the load target).
- The sound-event mapping (which .wav plays on footstep/swing/shot/pickup/
  hit/UI click lives in the client; `sounds/` holds only the assets). Every
  sound is original, generated by `tools/generate_sounds.cpp`; the per-file
  source/license listing is `sounds/CREDITS.md`, and sounds without a safe
  license are rejected.
- The builtin fallback copies of every definition in this folder.
