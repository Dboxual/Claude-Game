# The Ruins of Gloamwood — Vertical Slice Progress

## Technical audit

- Rendering: Three.js 0.166 ES modules with `WebGLRenderer`, soft shadow maps, ACES tone mapping, `EffectComposer`, and `UnrealBloomPass`.
- Delivery: static browser project served over HTTP; no build step or asset pipeline.
- Existing systems to preserve: gathering, inventory, crafting, mastery/progression, quests, factions, guilds, health/energy, combat, third/first-person camera, lock-on, XP orbs, loot, death, and respawn.
- Current visual limitation: nearly all world and actor art is created directly in `game3d.js` from basic geometry, with a single flat cylinder as terrain and randomized placement.
- Performance constraint: browser WebGL, so the slice will use shared materials/geometries, grouped authored placement, limited shadow casters/lights, capped particles, and modest segment counts.

## Implementation plan

1. Replace the flat arena with an original sculpted terrain mesh driven by a deterministic height function. Author a playable route through a forest threshold, raised path, stream crossing, ruined courtyard, raider camp, temple entrance, and boss arena.
2. Create a coherent articulated Wayfarer rig with clothing, armor, limbs, boots, hands, hair/hood, and a readable sword. Animate idle, locomotion, attacks, dodge, hit, death, and gathering through procedural rig poses.
3. Create two articulated enemies—Gloamwood Raider and Hollow Guardian—and a bespoke oversized Hollow Colossus boss with stone armor, asymmetrical proportions, weapon/attack silhouettes, and restrained weak-point glow.
4. Replace repeated cone trees and standalone architectural primitives with authored tree variants, roots, foliage clusters, stone courses, arches, stairs, collapsed walls, statues, rubble, and a distant landmark.
5. Improve combat presentation, XP motion/merging, rarity-aware loot, ambient sound layers, footsteps, camera readability, and UI hint fading without changing underlying game rules.
6. Validate syntax and runtime after every major phase. Complete Chromium interaction checks, measure frame rate, capture eight requested screenshots, and record a short gameplay video.

## Asset policy

This phase uses original procedural meshes, materials, and synthesized audio generated by project code. No external art, models, textures, music, or copyrighted game assets are planned.

## Focused correction pass

### Profiled causes

Controlled Chromium/SwiftShader variants identified the actual frame costs before changes:

- Baseline before correction: approximately 13.5 FPS, 290 draw calls, 73,682 triangles
- Scene inventory before correction: 297 meshes, 121 shadow casters, 27 materials, 3 lights
- Real-time shadows disabled in isolation: approximately 19 FPS
- Half-resolution rendering in isolation: approximately 19 FPS
- Authored world hidden: approximately 50 FPS
- Legacy residue found: 90 old scattered meshes plus legacy factories were still constructed outside the replacement world root
- Largest geometry costs: detail-1 foliage polyhedra and subdivided ruin blocks

### Corrections implemented

- Removed the legacy render root and its leaked scenery
- Replaced real-time shadow-map rendering with one instanced contact-shadow system
- Converted repeated grass, bushes, mushrooms, fallen branches, and combat sparks to `InstancedMesh`
- Reduced foliage and ruin-block topology while retaining silhouettes
- Batched static world geometry and each interactive resource node by material
- Replaced costly diffuse `MeshStandardMaterial` use with Lambert materials where metallic/roughness response was unnecessary
- Kept Standard materials for bronze, iron, magic, and other intentional highlights
- Disabled bloom; magic remains readable through restrained emissive materials
- Capped and pooled combat particles at 72
- Added distance culling and reduced update frequency for distant enemies
- Throttled minimap drawing to 5 Hz and camera collision raycasts to 10 Hz
- Reused temporary vectors and the collision raycaster inside the update loop
- Lowered internal browser render ratio to 0.5 while retaining 1440×900 interface/capture output
- Added visible first-person arms and weapon with attack motion
- Added distance-driven locomotion phase, dodge pose, gathering pose, hit reaction, hit stop, contact dust, impact sparks, and positioned damage numbers
- Replaced remaining cylindrical player/raider torso shells with shaped lathed clothing meshes

## Adaptive performance & touch pass — 2026-07-19

Goal: playable on ANY device (owner reported poor FPS on their hardware).

- Adaptive quality engine: rolling-FPS-driven render-scale between 0.3 and a
  device-based ceiling; evaluates every ~2s with hysteresis; if the floor still
  can't hold ~28 FPS the bloom composer is dropped and the scene renders direct
  ("PERFORMANCE MODE"); recovers upward when headroom returns.
- Touch devices start in performance mode at reduced scale by default
  (pointer:coarse detection).
- Discrepancy fixed: bloom was documented as disabled in the correction pass but
  was still active in game3d.js line 13 — now tier-controlled.
