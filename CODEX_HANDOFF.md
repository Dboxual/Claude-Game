# CODEX_HANDOFF — TacMove

**Read this first.** It is the single entry point for continuing this project.
`HANDOFF.md` is the deeper living status doc; `README.md` has long-form feature
notes; this file is the fast, structured orientation.

---

## 1. Current project goal

TacMove is a **cross-platform, OpenGL-3.3 game hub** ("game modes are the games
inside the game"). The long-term vision is a **social / action game hub** — one
day the kind of place VRChat is, but *not yet*. The near-term goal is narrow and
concrete:

> **Ship one polished, great-feeling playable vertical slice: MiniCS — a neon
> arena shooter with a real round loop, satisfying gunplay, and a look that
> reads as a real game, not a blueprint.**

Everything else (multiplayer, other modes, social features) is deliberately on
hold. Do **not** chase VRChat features, add multiplayer, add new game modes, or
add blank menus yet. Make MiniCS *feel good*, then broaden.

## 2. Current branch / state

- Branch: **`wip-m3-content-platform`** (the active WIP; **not** merged to
  `main`). Continue here.
- Builds clean on Windows/macOS/Linux, OpenGL 3.3, **zero warnings** (`/W4`,
  `-Wall -Wextra`). Client + dedicated server both compile.
- MiniCS is a genuine, verified vertical slice (round loop + gunplay + neon
  look). The other five modes still boot into a real world but are "Early".

## 3. How to build

First build bootstraps vcpkg + compiles SDL3 from source (a few minutes once).

- **Windows:** `run_windows.bat` (needs VS 2022 Build Tools + C++ workload), or
  directly: `cmake --preset desktop && cmake --build build --config Release`.
  Output: `build/Release/tacmove.exe` (+ `tacmove_server.exe`).
- **macOS:** `./run_macos.sh` (Xcode Command Line Tools).
- **Linux:** `./run_linux.sh` (apt deps in `README.md`).

The build copies `config/` and `server/` next to the binary automatically, so
the exe finds movement config, content defs, and sounds wherever it launches.

## 4. How to run

- Run the binary (`build/Release/tacmove.exe`, or the `run_*` script). You land
  on the **main menu**, which orbits the neon MiniCS arena as its backdrop.
- Controls in a gun mode: `WASD` move, mouse aim, `LMB` fire, `R` reload, `ESC`
  pause/menu.

## 5. How to launch MiniCS

- **From the menu:** `PLAY → GAME MODES → MINICS`.
- **Directly (dev):** `build/Release/tacmove.exe --mode minics`.
- A round: spawn south → ~2 s `GET READY` countdown → 75 s firefight vs 5 ranged
  bots → `VICTORY`/`DEFEAT` end screen → press fire/space to play again (`ESC`
  for menu). Kill all five before the clock runs out.

## 6. What systems currently exist

Layered so gameplay stays portable (enforced by CMake target boundaries):

- **`src/shared/`** (`tac_shared`, std+glm only, compiled into client *and*
  server): `player` (fixed-tick movement sim), `world` (box collision +
  raycasts + entities), `world_template` (map geometry, incl. neon arena),
  `content` + `content_loader` (weapon/entity defs, code + `.cfg`), `items`,
  `crafting`, `melee` (Chivalry-style combat resolver), `game_mode` (mode
  registry), `config`, `world_save`, `log`.
- **`src/client/`**: `game.{h,cpp}` — the whole client: menu state machine,
  **MiniCS round controller + enemy AI**, combat, viewmodel, HUD, settings.
  `platform/` (SDL3 window/input/audio/fs seam), `renderer/` (`IRenderer` +
  the **OpenGL 3.3** backend; vulkan/metal/gles are honest 9-line stubs).
- **`src/server/`**: `tacmove_server`, a headless config-only tick loop
  (does not yet simulate combat or load content — reserved for networking).
- **Audio**: original synthesized (CC0) one-shots generated by
  `tools/generate_sounds.cpp` into `server/content/sounds/`.

## 7. What was recently added (this pass + the one before it)

**MiniCS vertical slice** (previous session):
- Round loop (countdown → live → win/lose → restart), lose-on-death.
- **Ranged enemy AI** (`minics_bot`): Advance→Strafe→Aim→Fire→Flinch, red
  tracers, distance-based accuracy, ~3-shot kill, animated (walk/aim/shoot/
  flinch), dies with a spark burst.
- **Glock feel**: magazine + reserve, `R` reload (with animation), dry-fire,
  and a recovering recoil view-punch.
- **Feedback + HUD**: hit + kill markers, `+100` score popups, kill-confirm /
  round-win / round-lose / reload / dry-fire / enemy-shot sounds, directional
  damage indicator, ammo / clock / enemies / score HUD.
- Deterministic `--frames` runs (fixed 60 Hz sim clock) for reproducible
  screenshots/smoke tests (`main.cpp`).

**Neon visual identity** (this pass):
- **Emissive rendering**: `BoxDraw`, `WorldBox`, and `VisualPart` gained an
  `emissive` field; the world shader now takes per-frame fog color/scale/max and
  emissive wins over shading + fog. `RenderFrame` carries per-mode
  `skyTop/skyHorizon/fogColor/fogScale/fogMax`.
- **MiniCS = dark neon dusk**: charcoal arena shell, cyan wall-top LED strips,
  magenta/cyan floor lane lines, an amber glowing platform rim, a lime start
  line, and a `neon_pillar` prop (glowing landmark/cover). Per-mode atmosphere
  set in `buildRenderFrame` (only `minics_arena`; every other world stays
  daylight and unchanged).
- **Menu background** now orbits the neon arena, so the hub looks real on open.
- **Punchier muzzle flash** (multi-petal star + white-hot core that flares then
  collapses).

## 8. What still feels bad (honest)

- **Enemy movement is still simple** — they now use cheap flank-lane pressure
  and can punish a spawn camper, but there is no true pathfinding, cover
  selection, or tactical squad behavior yet.
- **Enemy fire is hitscan with distance-rolled accuracy** — no leading, no
  suppression, so incoming damage can feel a little random.
- **Headshots exist but are simple** — a head sub-hitbox gives bonus damage and
  a distinct cue, but there is still no detailed per-limb hit model.
- **Enemy death is a short scripted topple**, not a real ragdoll or persistent
  body.
- Everything is **box-art** (world, enemies, viewmodel). No textures/models.
- Muzzle flash is bright but still a blocky composite; tracers are box-composed.
- The five non-MiniCS modes are **Early** (real world, no ruleset depth).

## 9. What Codex should work on next (direction)

Keep the scope tight: **push MiniCS from "good vertical slice" to "polished neon
action shooter you want to replay."** In order of impact:

1. Better enemy movement/AI (flanking, using lanes/cover intentionally).
2. Headshot hitboxes + a satisfying death reaction.
3. Stronger gun feel (ADS/aim-down-sights, a subtle hipfire spread, weightier
   recoil pattern, screen-space muzzle light).
4. More neon life: animated emissive pulsing, a couple more props, particle
   polish on impacts/kills, a scoreboard/round-summary.
5. Then general cleanup/optimization (see §15) — but **not** a rewrite.

Do **not**: add multiplayer, add modes, add blank menus, replace the renderer,
or start chasing social/VRChat features.

## 11. Which files are safe to edit freely

- `server/content/**/*.cfg` — weapon/entity/def tuning (hot-swappable data).
- `src/shared/world_template.cpp` — map layouts, neon accents, colors.
- The MiniCS controller / HUD / viewmodel / neon code in `src/client/game.cpp`
  (all gated on `minicsActive_` / `currentWorldType_`, so changes stay scoped).
- `tools/generate_sounds.cpp` — audio (deterministic; re-run to regenerate).
- Tuning constants in `game.{h,cpp}` (see the tuning index).
- Docs (`CODEX_HANDOFF.md`, `HANDOFF.md`, `README.md`).

## 12. Which files should be treated carefully

- `src/shared/player.cpp` — the movement sim (accel/friction/air-strafe). It is
  tuned and feels good; change deliberately, verify feel, don't casually refactor.
- `src/shared/melee.cpp` — the shared combat resolver used by the duel bots /
  sandbox; MiniCS doesn't use it, but the sandbox does.
- `src/client/renderer/render_types.h` — a data contract consumed by all
  renderer backends; adding fields is fine (defaults), reordering/removing is not.
- `src/client/main.cpp` — the composition root + fixed-tick loop + dev flags.
- The fixed-tick block inside `Game::update()` (the `accumulator_` loop).
- `CMakeLists.txt` / target boundaries — they enforce the portability seams
  (shared has no SDL/GL; server links only shared). Keep that invariant.

## 13. Known bugs / rough spots

- Enemies can still occasionally bunch or oscillate against cover; lane routing
  helps, but it is not full pathfinding.
- The MiniCS enemy-health readout is hidden now; keep future target info
  minimal so the crosshair stays clean during combat.
- The menu backdrop shows the arena's template `duel_bot`s standing idle
  (cosmetic only; MiniCS itself uses `minics_bot`).
- Offscreen/background frame rate varies wildly (SDL throttles unfocused
  windows) — only `--frames` runs are fixed-step, so use those for captures.
- No crashes known: all six modes run 200 frames and exit 0; server runs clean.

## 14. Rendering limitations (stay inside these)

- **OpenGL 3.3 core only.** No ray tracing, PBR, compute, geometry/tessellation
  shaders, MSAA config, or platform-specific paths. Vulkan/Metal/GLES are stubs.
- **Box geometry only** — one unit cube, instanced per draw via a model matrix.
  No textures, no meshes, no skeletal animation. All "models" are box composites.
- **Emissive = full-bright**, not real bloom/HDR. Neon reads by dark-vs-bright
  contrast + fog, not by a glow post-process.
- **Lighting** is a single hard-coded directional term + a checker floor tint.
- **Fog** is linear camera-distance in the world fragment shader.
- Screenshots are 24-bit **BMP** (`--screenshot`), no PNG encoder in-repo.
- Prefer: simple geometry, strong/emissive colors, fog, gradients, particles
  (box sparks), and clean 2D UI (rects + a 5x7 bitmap font). That palette is
  enough to look intentional — lean into it rather than fighting the renderer.

## 15. Optimization notes

Current scale is tiny, so nothing is urgent, but for reference:
- **Draw calls**: one `glDrawArrays(36)` per box (world boxes, entity parts,
  viewmodel, sparks, tracers). Fine at current counts (low hundreds). The first
  real optimization if counts grow is **instanced rendering** (one draw per
  color/material, per-instance model matrices) — the data is already box-only.
- **Entity lookups**: `World::entityById` and the MiniCS loops are linear scans
  over `entities_`. Fine for dozens; revisit if entity counts explode.
- **No frustum culling / spatial partition** — every box is submitted each
  frame. Acceptable for one small arena.
- `game.cpp` is large (~3.7k lines). Extracting the menu-model and the MiniCS
  controller into their own units would help *before* the file grows further —
  but do it as a careful move, not a rewrite, and only if it earns its keep.

### Fast reference: dev flags (client)

`--mode <id>` boot a mode · `--menu <screen>` boot a menu screen ·
`--world` sandbox · `--equip <id>` start holding a weapon ·
`--frames N` exit after N (fixed 60 Hz) · `--fire-at N` inject one shot ·
`--screenshot <path.bmp>` dump the last frame · `--novsync` uncap · `--paused`.

Example neon-fight capture:
`build/Release/tacmove --mode minics --fire-at 396 --frames 400 --screenshot shot.bmp`
