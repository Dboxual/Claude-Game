TacMove — WIP status (branch wip-m3-content-platform, 2026-07-06, update 8)
============================================================================

Eighth drop: WORLD TYPES foundation. Create World can now build different
kinds of local worlds from starting templates. Deliberately small: no
networking, no map file loader, no procedural terrain, no combat/renderer
changes. The polished test glade (Start Test World) is untouched.

DONE THIS DROP (built on Windows x64, client + server, /W4 zero warnings)
--------------------------------------------------------------------------
NEW - WORLD TEMPLATE SYSTEM (shared, server-safe):
- src/shared/world_template.{h,cpp}: a WorldTemplate is pure data - id,
  display name, description, static geometry (vector<WorldBox>), spawn
  point, and starting entity placements (defId + pos + respawn, resolved
  against the content registry by the caller, exactly like a save file's
  entities). std + glm only; no IO, no content lookups. Shaped so a future
  loader can fill it from server/content/maps/*.cfg without touching callers.
- Five built-in types (worldgen::templates(), in menu order):
    flat_sandbox    - empty enclosed 40x40 flat map (the default + fallback)
    duel_yard       - small arena, sword + shield + duelist bot + dummy
    aim_range       - 12x60 lane, Glock at the line, 5 dummies down-range
    movement_course - jump steps, crouch tunnel, strafe pillars (geometry only)
    social_hub      - open plaza, low dais/benches, a few crates/barrels
- World::buildFromTemplate(t): swaps geometry + spawn point, clears
  entities. buildFlatMap() was removed - flat_sandbox replaces it (single
  source of truth for the flat map). buildTestMap() (the glade) is untouched.

SAVE FORMAT v3 (backward compatible):
- world.cfg gains one line: `world_type = <id>`. WorldSaveData grew a
  worldType field; serialize writes it, parse reads it.
- Old saves without world_type parse fine and load as flat_sandbox. v1/v2
  entity lines still parse. Unknown/newer world_type ids fall back to
  flat_sandbox with a logged warning, never a failure.

CREATE WORLD UI:
- The Create World screen now has a row of world-type tabs (FLAT / DUEL /
  AIM / MOVEMENT / SOCIAL) under the name box, highlighting the selected
  one, with the type's description shown live below. Enter or CREATE builds
  the chosen type, places its starting entities, and saves world_type.
- Load World is unchanged in use: it reads world_type and rebuilds the
  right geometry. Start Test World is still the fixed glade, never saved.

VERIFIED THIS DROP
------------------
- Clean /W4 Release build, client + server, zero warnings.
- Smoke: client --world --frames 240 --novsync exit 0; server --ticks 256
  exit 0.
- Throwaway unit check against tac_shared (scratch, not committed): all 5

  world_type at version 3; a legacy v2 save (no world_type) parses with an

WHERE THE NUMBERS LIVE (for taste tuning)
-----------------------------------------
- All world-type geometry + placements: src/shared/world_template.cpp (one
  function per template; helpers box()/enclose() at the top).
- Save format: src/shared/world_save.cpp (serialize/parse).
- Create World menu: buildMenuLayout() CreateWorld case + the CreateWorld
  block in appendMenuDraws(), both in game.cpp.

UNFINISHED (known, deliberate)
------------------------------
- No maps/*.cfg loader yet: templates are code-defined. The struct is the
  load target when that lands.
- Templates place only existing content ids (sword/shield/glock/duel_bot/
  training_dummy/crate/barrel). No new content was added.
- Movement course has no timer/checkpoints; it is layout only, as asked.
- Social hub is a local world type only - no multiplayer, nothing faked.
- The dedicated server still does not build worlds from templates (it is
  config-only until the networking/sim milestone); the template module is
  shared so it can when that lands.

BROKEN / WATCH OUT
------------------
- World::buildFlatMap() was removed. Any out-of-tree caller should use
  buildFromTemplate(worldgen::fallbackTemplate()) instead.
- world.h forward-declares WorldTemplate; world.cpp includes
  world_template.h. Keep that direction (no include cycle).

TacMove — WIP status (branch wip-m3-content-platform, 2026-07-06, update 7)
============================================================================

Seventh drop: FEEL/FIX pass on the playtest complaints. No new systems
except player health (which the playtest explicitly asked for); everything
else is fixing what already existed: tracers, gun audio, animation
readability, HUD. First drop built AND visually verified on macOS
(Apple M2, OpenGL backend) via a new automated screenshot hook.

DONE THIS DROP (built on macOS arm64, client + server, zero warnings)
--------------------------------------------------------------------------
FIXED - GUN AUDIO (the real bug): SdlAudio dropped a sound when all 8
  voices were busy. Footsteps + rapid shots + impact sounds starve 8
  voices easily, so gunshots randomly vanished. Now 16 voices and, when
  full, the OLDEST voice is stolen round-robin (SDL_ClearAudioStream)
  instead of dropping the new sound. A cut footstep tail is inaudible; a
  swallowed gunshot was the "gun sounds missing" report. Volumes still
  applied per-play as master*sfx from the settings screen.
  Dry-fire click NOT added: there is no ammo system, so nothing can dry-fire.

FIXED - TRACERS: were a full-length instant line from a rough eye offset
  (they smeared across the screen for one frame and read as broken). Now:
  a short bright streak (max 2.4 m) that leaves the muzzle position and
  TRAVELS to the impact point at ~130 m/s visible speed; only the
  traveled part is drawn, thick enough (4.5 cm) to survive projection
  when flying nearly parallel to your own view ray. The impact puff
  (sparks + a bright flash) pops when the tracer ARRIVES, exactly where
  the shot landed. Verified frame-by-frame with screenshots: muzzle
  flash on the shot frame, amber dash converging on the crosshair
  mid-flight, impact flash dead-center on arrival at a 13 m wall.

NEW - PLAYER HEALTH + HEALTH BAR: duelist bot Hit/Interrupted strikes now
  deal real damage (melee::damageOf over a 12 base, so ~12-14 per hit).
  Clean bar bottom-left: color walks green -> amber -> red, a ghost bar
  visibly drains after damage (the shown value eases toward the real
  one), the fill flashes white on the hit frame, number next to it.
  Slow regen (8/s) after 5 s out of trouble keeps the sandbox friendly.
  Going down is soft: respawn at spawn with full health, weapons kept,
  carried prop released + settled, big flash/shake + toast. The old red
  damage vignette still layers on top.

ANIMATION READABILITY:
  - Fists finally BOX: a left slash is thrown by the LEFT fist (the
    other stays in guard); everything else stays on the right. The old
    everything-on-one-fist version was most of "melee looks weird".
  - Active-phase sweep uses smoothstep, not linear - the blade
    accelerates out of the windup and decelerates into the strike point.
  - Stab pulls to the hip and thrusts visibly deeper (-0.46 vs -0.36);
    jab pops further; kick boot extension smoothstepped.
  - Glock recoil: quadratic snap-decay on the muzzle rise (17 deg), a
    touch of roll, straight-back shove - fast up, quick settle.

ITEM/WORLD READABILITY: weapon pickups now hover AND slowly spin
  (world-space oriented boxes) - they read as items, not stray geometry.
  Bot target readout lost its debug-y stamina number (name + health +
  state words stay).

HUD CLEANUP: crosshair is four drawn arms with a center gap (rects, not
  a font "+"). Default screen shows ONLY: crosshair, health bar, stamina
  bar (melee in hand only), weapon slots, interaction prompts, toasts,
  the 8 s spawn hint. Debug HUD stays F1. NOTE: the local settings.cfg
  had show_debug_hud=1 left over from an old default - that is why the
  playtest saw debug spam. Flipped to 0 locally; fresh installs were
  already clean.

NEW DEV TOOLING (small, permanent, this is how this drop was verified):
  - --screenshot <path>: dumps the last rendered frame as a BMP
    (glReadPixels in the GL backend; IRenderer::requestScreenshot is a
    default-no-op for other backends). Works headless-ish with --frames.
  - --fire-at N: injects one synthetic attack press at frame N, so an
    automated run can capture recoil/tracers/swings mid-flight.
  - gl_loader grew glReadPixels + GL_PACK_ALIGNMENT/GL_RGBA.

WHERE THE NUMBERS LIVE (for taste tuning)
-----------------------------------------
- Tracer speed/thickness/streak: tryAttack (spawn) + the Tracer block in
  buildRenderFrame (draw) in game.cpp.

UNFINISHED (known, deliberate)
------------------------------
- Bots still cannot die "at" you - they have no damage numbers of their
  own beyond the existing health bar; no death animation, they just pop.
- Tracer origin is an approximation of the on-screen muzzle (viewmodel
  renders through its own FOV projection; exact match is impossible).
- No ammo, so no reload or dry-fire. Music slider still drives nothing.
- Viewmodels are still box-art. Better proportions, but they will look
  placeholder until a real art pipeline exists.
- --fire-at only injects ATTACK; kicks/blocks still need a human.

BROKEN / WATCH OUT
------------------
- IRenderer grew requestScreenshot (virtual, default no-op) - out-of-tree
  backends recompile fine but ignore it.
- config/settings.cfg is local/gitignored: this drop set show_debug_hud=0
  in YOUR copy. F1 brings it back any time.
- Player health lives in the client (playerHealth_ in game.cpp), NOT in
  shared/ - when the server milestone lands it must move server-side.
